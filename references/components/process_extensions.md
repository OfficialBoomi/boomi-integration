# Process Extensions (Environment Overrides)

Environment extensions make connection settings, operation settings, and Dynamic Process Properties (DPPs) configurable per-environment without modifying process XML. They are defined in the `<bns:processOverrides>` element at the end of a process component's XML.

## Contents

- Operation Overrides

## processOverrides XML Structure

The `<Overrides>` element lives inside `<bns:processOverrides>` and uses an empty namespace (`xmlns=""`). Only sections with actual content need to be present.

**Minimal example** (DPP-only):
```xml
<bns:processOverrides>
  <Overrides xmlns="">
    <Properties>
      <PropertyOverride name="DPP_MY_SETTING"/>
    </Properties>
  </Overrides>
</bns:processOverrides>
```

**Full structure** (as generated by Boomi GUI):
```xml
<bns:processOverrides>
  <Overrides xmlns="">
    <Connections>...</Connections>
    <Operations>...</Operations>
    <PartnerOverrides/>
    <Properties>...</Properties>
    <Extensions>
      <ObjectDefinitions><unusedProfiles/></ObjectDefinitions>
      <DataMaps><unusedMaps/></DataMaps>
    </Extensions>
    <CrossReferenceOverrides/>
    <PGPOverrides/>
    <DefinedProcessPropertyOverrides/>
  </Overrides>
</bns:processOverrides>
```

The platform stores exactly what is pushed and does not auto-complete missing sections.

**WARNING:** Deploying a process with empty `<bns:processOverrides/>` or empty `<Overrides xmlns=""/>` actively **removes** that process's extension declarations from the environment. Extension values previously set via the API are lost. This is destructive. See references/guides/boomi_error_reference.md Issue #23.

## Connection Overrides

List each connection by component ID, with overrideable fields specified. Only fields you want to be environment-configurable need to be listed.

```xml
<Connections>
  <ConnectionOverride id="c7d489dc-0818-4bfa-9d4b-c3e75a4f8e74">
    <field id="url" label="URL" overrideable="true"/>
  </ConnectionOverride>
</Connections>
```

- `id` on `ConnectionOverride` = the connection's component ID (must reference a real platform component)
- `field id` = the field identifier from the connection's configuration
- `label` = human-readable name shown in the Boomi GUI extensions tab
- `overrideable="true"` = configurable per-environment

The full field enumeration pattern (listing all fields with `overrideable="false"` on those you don't want configurable) is a GUI convention, not a requirement. List only what you want overrideable.

## Operation Overrides

Same pattern as connections -- reference by operation component ID, list only the fields you want overrideable.

```xml
<Operations>
  <OperationOverride id="177e7603-38c3-49bb-b083-df2fc71459b3">
    <field id="toolName" label="Tool Name" overrideable="true"/>
  </OperationOverride>
</Operations>
```

## Which Connections/Operations to List

Only list connections and operations actually used by steps in the process. The platform does not enforce this -- omitting a used connection simply means it uses design-time values and is not environment-configurable.

## Dynamic Process Property Overrides

DPPs are made extendable by listing them as `<PropertyOverride>` entries in the `<Properties>` section. No `overrideable` attribute exists -- the platform schema actively rejects it. Presence alone controls extendability.

```xml
<Properties>
  <PropertyOverride name="DPP_SF_AGENT_ID"/>
  <PropertyOverride name="DPP_SF_CLIENT_ID"/>
</Properties>
```

Each entry is independently controllable: adding or removing a single entry affects only that DPP. DPPs used in the process but not listed in `<Properties>` are not exposed to environment extensions.

### Reading DPP Values at Runtime

DPPs must always be read using `valueType="process"` + `<processparameter>`. Using `valueType="track"` silently returns empty string. See references/guides/boomi_error_reference.md Issue #24 for correct/wrong XML patterns.

**Set Properties interaction:** A Set Properties step that explicitly sets a DPP overwrites the extension-injected value for the remainder of that execution. Extension values are initial values at process start; in-process Set Properties steps take precedence.

## Extensions Section (ObjectDefinitions)

The `<Extensions>` section is entirely optional and has no effect on runtime behavior, connection overrides, or the Environment Extensions API. It contains GUI-generated metadata about connector types and profile associations.

**For programmatic development, omit the `<Extensions>` section entirely.** Connection overrides in `<Connections>` and property overrides in `<Properties>` are what drive extension behavior.

If included for any reason, it must contain both `<ObjectDefinitions>` and `<DataMaps>` children -- a self-closing `<Extensions/>` is rejected by schema validation. Minimum valid content:

```xml
<Extensions>
  <ObjectDefinitions><unusedProfiles/></ObjectDefinitions>
  <DataMaps><unusedMaps/></DataMaps>
</Extensions>
```

## Environment Extensions API

Extension values are read and updated at the **environment** level:

```
GET  /api/rest/v1/{accountId}/EnvironmentExtensions/{environmentId}
POST /api/rest/v1/{accountId}/EnvironmentExtensions/{environmentId}
```

A `{componentId}` path segment can be appended but is ignored -- both endpoints always operate at the environment level, returning/updating all extensions across all deployed processes in that environment.

### GET behavior
- Returns JSON with `connections` (all overrideable fields across all deployed processes) and `properties` (all extendable DPPs)
- Only fields marked `overrideable="true"` appear
- DPP entries include `name` and optionally `value` (absent when no extension value has been set)

### POST behavior
- Accepts a partial payload -- only properties/connections included are updated; others are unaffected
- Response echoes back only the submitted data, not the full environment state
- Values take effect after redeployment of the process

### PUT
PUT returns `405 Method Not Allowed` -- not supported for this resource.
